Footer Element Accessibility Misuse Examples

Issue 1: Using footer for main content

Website Title

About Our Company

This is the main content about our company, but it's incorrectly placed in a footer element.

Our company was founded in 2005 with the mission to provide excellent service...

Our Services

  • Service 1
  • Service 2
  • Service 3

Problem: The footer element is being used to contain main content rather than supplementary information. This misuse confuses screen reader users who expect footers to contain information like copyright, contact info, or links.

Issue 2: Using div with styling instead of semantic footer

Website Title

Main content goes here...

© 2023 Company Name

Contact: info@example.com

Problem: Using a div with visual styling instead of a semantic footer element deprives screen reader users of the landmark navigation that footers provide. Screen readers can't identify this as footer content.

Issue 3: Footer with incorrect ARIA role

Website Title

Main content goes here...

© 2023 Company Name

Contact: info@example.com

Problem: Applying the "main" role to a footer creates confusing and contradictory semantics. This tells assistive technologies that this content is both footer content and the main content of the page.

Issue 4: Missing heading in a complex footer

Email: info@example.com

Quick Links

  • Home

© 2023 Company Name

Problem: Complex footers with multiple sections should use proper headings (h2-h6) for each section. Using paragraph tags for section titles makes it difficult for screen reader users to navigate the footer sections.

Issue 5: Footer inside a section without sectioning context

Website Title

This is some content in a generic div...

This footer relates to the div above, but the relationship is unclear.

Problem: When a footer is placed inside a generic container like a div, the relationship between the footer and its parent content is not semantically clear. Footers should be inside sectioning elements (article, section, aside, nav) or at the document level.

Issue 6: Multiple footers with contentinfo role at document level

Problem: Multiple footers at the document level both with contentinfo role creates confusion for screen reader users. The contentinfo role (which is implicit for footer elements at the document level) should be used only once per page.

Accessibility Best Practices for the Footer Element

  • Use the <footer> element for supplementary information about its nearest sectioning ancestor
  • Ensure document-level footers contain information like copyright, contact info, or secondary navigation
  • Use proper headings (h2-h6) for distinct sections within complex footers
  • Avoid using the <footer> element for main content
  • Place footers inside sectioning elements (article, section) when the footer relates specifically to that content
  • Do not override the native semantics of footer with inappropriate ARIA roles
  • Ensure only one footer at the document level has the contentinfo role (this is implicit for document-level footers)
  • Don't use empty footers just for visual spacing
  • Use the <address> element appropriately within footers for contact information
